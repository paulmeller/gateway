# v0.5.0 ‚Äî Tenancy + multi-provider ops

> Scope: **Option B** from the v0.5 scoping chat ‚Äî tenancy + adjacent wins.
> If this bloats past ~2 weeks of work, drop to Option A (tenancy-only).
> If nobody's actually asked for tenancy by the time we start, drop to
> Option C (ops hardening only). Left unresolved on purpose ‚Äî ship order
> below is modular.

## Context

v0.4 shipped virtual keys, budgets, rate limits, session fallback,
per-key cost dashboard, and the Anthropic-only upstream pool. The
codebase is ready for genuine multi-tenant use but scope ends at a
single-tenant gateway: there is no `tenant_id` enforcement anywhere,
all admins are global, and the upstream pool is single-provider.

Three pressures point toward v0.5 being "the release that makes this
usable by teams":

1. **Architect-approved tenant design is already on file** ‚Äî v0.4's
   nullable `tenant_id` on `api_keys` is an explicit reservation. The
   full ~5‚Äì8-day delivery was held back intentionally to avoid cramming
   it into v0.4. v0.5 is the natural landing zone.
2. **Upstream pool is Anthropic-only.** OpenAI/Gemini customers can't
   use the rotation + auto-disable features yet. Symmetrical extension
   unlocks both engines.
3. **Operational hardening** (webhook signatures, Redis-backed rate
   limiting, audit log) has been bubbling since early reviews. Each is
   ‚â§1 day individually but matters for hosted deployments.

## Non-goals

- OpenAI-compatible `/v1/chat/completions` ‚Äî LiteLLM's turf. Don't dilute
  the "Anthropic Managed Agents compatible" positioning.
- SSO / SAML / OIDC ‚Äî real SaaS authentication is v1.0 scope. Bearer
  tokens with tenant scope is a legitimate stepping stone.
- Prompt management / versioning ‚Äî separate product category.
- Turn-level caching ‚Äî still wrong for agents with side-effectful tools.

## Design decisions (resolved)

| # | Decision | Rationale |
|---|---|---|
| 1 | **Full tenant isolation, not a half-ship.** Every list/get/delete handler gains `WHERE tenant_id = ?` for non-global-admin callers. CREATE stamps tenant from `ctx.auth.tenantId`. | Architect: "a column without enforcement is a lie that future maintainers will trust." Half-shipping is worse than not shipping. |
| 2 | **3-tier admin.** Global admin (tenantId=null, admin=true) ‚Üî tenant admin (tenantId=T, admin=true) ‚Üî tenant user (tenantId=T, admin=false, scope=‚Ä¶). Two tiers can't bootstrap a second tenant. | The second-tenant paradox ‚Äî someone must be outside the tenant system to create tenants. |
| 3 | **Tenant from key, not header.** `AuthContext.tenantId` is read from the API key row. One key, one tenant. No `X-Tenant-Id` header. | Adding a header is a footgun that buys flexibility nobody has asked for. |
| 4 | **Explicit migration command.** `gateway tenants migrate-legacy` is one-shot, interactive, opt-in. Without it, seed key stays global admin and legacy rows stay `tenant_id=null`. | Silent data walls on upgrade = churn. The operator must see the transition. |
| 5 | **Fallback cross-tenant validation.** `agents.fallback_json` target must be same-tenant as the source agent. Validated at save (in `handleUpdateAgent`) AND at runtime in the fallback loop (belt-and-suspenders against post-save reassignment). | Otherwise tenant X could leak traffic to tenant Y's agents via a stale fallback reference. |
| 6 | **Upstream pool stays global in v0.5.** Per-tenant pools are deferred. One global pool, operator adds all keys. | Operational simplicity for v0.5. A per-tenant pool adds complexity that most self-hosted installs don't need ‚Äî revisit in v0.6 when someone asks. |
| 7 | **Resources explicitly out of tenant model (v0.5).** `memory_stores`, `files`, `settings`, `skills_catalog`, `events` (inherit from session), `proxy_resources`, `anthropic_sync` stay global with a documented note. | Per the architect: each needs a deliberate decision. Shipping "partly tenant-aware" adds confusion; shipping "tenant-aware everywhere" is its own 3-day subtask. v0.5 does the big resources (agents/envs/vaults/sessions); follow-up release covers the rest. |
| 8 | **Rate limit store ‚Üí Redis option.** Default remains in-memory; operators who cluster set `RATE_LIMIT_BACKEND=redis REDIS_URL=...`. | Matches the plan's "documented caveat" promise. Redis is standard-issue for the target audience. |
| 9 | **Webhook signatures required by default.** New webhooks get a generated HMAC secret; signatures use `sha256` in `X-AgentStep-Signature`. Existing webhooks auto-opt-in at migration ‚Äî callers without signature verification won't break (we don't check the other direction), but new clients get to verify. | Security defaults matter. Stripe/GitHub conventions are well-known. |
| 10 | **Audit log is in-DB, not external.** `audit_log` table: who (key_id), what (event type), which tenant, when, optional ref (resource id). Appended on every admin action. Queryable via `/v1/audit` admin endpoint. | Self-hosted audience benefits from a log colocated with their data. External SIEM integration is a v0.6+ concern. |

## Scope (4 PRs in dependency order)

### PR1 ‚Äî Tenancy foundation

**Schema:**
```sql
CREATE TABLE IF NOT EXISTS tenants (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  archived_at INTEGER
);
ALTER TABLE agents       ADD COLUMN tenant_id TEXT;
ALTER TABLE environments ADD COLUMN tenant_id TEXT;
ALTER TABLE vaults       ADD COLUMN tenant_id TEXT;
ALTER TABLE sessions     ADD COLUMN tenant_id TEXT;
-- api_keys.tenant_id already exists (reserved in v0.4)
CREATE INDEX idx_agents_tenant       ON agents(tenant_id);
CREATE INDEX idx_environments_tenant ON environments(tenant_id);
CREATE INDEX idx_vaults_tenant       ON vaults(tenant_id);
CREATE INDEX idx_sessions_tenant     ON sessions(tenant_id);
CREATE INDEX idx_api_keys_tenant     ON api_keys(tenant_id);
```

All idempotent via the existing `PRAGMA table_info` pattern.

**Files to modify:**

- `packages/agent-sdk/src/db/migrations.ts` ‚Äî add columns + `tenants` table + indexes
- `packages/agent-sdk/src/db/tenants.ts` (new) ‚Äî `createTenant`, `listTenants`, `getTenant`, `archiveTenant`
- `packages/agent-sdk/src/types.ts` ‚Äî new `Tenant`, `TenantRow` interfaces; `AuthContext` gains `isGlobalAdmin: boolean`
- `packages/agent-sdk/src/auth/middleware.ts` ‚Äî compute `isGlobalAdmin = tenantId === null && admin` from the row
- `packages/agent-sdk/src/auth/scope.ts` ‚Äî new `tenantFilter(auth)` helper returning either `null` (global admin = no filter) or the tenant id string
- `packages/agent-sdk/src/handlers/tenants.ts` (new) ‚Äî admin CRUD: `POST/GET/PATCH/DELETE /v1/tenants`
- `packages/agent-sdk/src/init.ts` ‚Äî seed a `default` tenant row on first boot (idempotent via `INSERT OR IGNORE`)

**Handler updates** ‚Äî filter injection on list/get + stamping on create:

- `handlers/agents.ts` ‚Äî 5 endpoints (list, get, create, update, archive)
- `handlers/environments.ts` ‚Äî 6 endpoints
- `handlers/vaults.ts` ‚Äî 8 endpoints (vault CRUD + 4 entry CRUD)
- `handlers/sessions.ts` ‚Äî 6 endpoints (list, get, create, update, archive, delete)
- `handlers/events.ts` ‚Äî list + POST (filtered by parent session's tenant)
- `handlers/threads.ts` ‚Äî list (filters via parent session)
- `handlers/metrics.ts` ‚Äî WHERE injection on SQL queries
- `handlers/api_keys.ts` ‚Äî list CRUD is tenant-scoped for tenant admins; create stamps tenant from auth
- `handlers/stream.ts` ‚Äî session fetch checks tenant
- `handlers/resources.ts` ‚Äî tenant-scoped via parent session

Pattern:
```ts
const tenantFilter = tenantFilter(auth);
const rows = tenantFilter
  ? db.prepare(`SELECT * FROM agents WHERE tenant_id = ? AND archived_at IS NULL`).all(tenantFilter)
  : db.prepare(`SELECT * FROM agents WHERE archived_at IS NULL`).all();
```

**Fallback same-tenant validation** (touches PR3 code):

- `handlers/agents.ts` update/create: parse `fallback_json`, fetch each referenced agent, verify same `tenant_id` ‚Äî reject with 400 if cross-tenant.
- `handlers/sessions.ts` fallback loop: before trying a candidate, fetch the candidate agent and assert `agent.tenant_id === session.tenant_id` ‚Äî skip candidate if mismatched (defense in depth).

**Migration CLI** (`packages/gateway/src/commands/tenants.ts`, new):

- `gateway tenants list`
- `gateway tenants create <name>`
- `gateway tenants migrate-legacy` ‚Äî interactive:
  1. Prompt to confirm (lists count of null-tenant rows across each table)
  2. Prompt for target tenant (default or create new)
  3. UPDATE every null-tenant row to the selected tenant in a single transaction
  4. Does NOT touch the seed key (it stays global admin) unless explicitly requested
- `gateway tenants archive <id>` ‚Äî soft-delete

**UI** (`packages/gateway-ui`):

- New `app-store` state: `selectedTenantId` (localStorage-persisted for global admins)
- `TenantSwitcher` component in sidebar header ‚Äî dropdown for global admins; static chip for tenant admins/users
- New `/settings/tenants` route + page with list + create + archive (global admin only)
- `ApiKeysPage` create-dialog: tenant selector when caller is global admin
- Every list hook (`use-agents`, `use-sessions`, etc.) ‚Äî server does the filtering; client needs no changes except surfacing tenant name in list columns where useful

**Tests:**
- `test/tenants-handlers.test.ts` ‚Äî CRUD, seed behavior, cross-tenant access denial, global admin visibility
- `test/tenancy-enforcement.test.ts` ‚Äî tenant X cannot list/get/delete tenant Y's agents/envs/vaults/sessions; 404 (not 403) for cross-tenant GET-by-id
- `test/fallback-cross-tenant.test.ts` ‚Äî save-time rejection, runtime skip of stale cross-tenant references
- `test/migrate-legacy.test.ts` ‚Äî full migration in one transaction

**Docs:**
- `docs/tenants.md` ‚Äî model, seed tenant, migration path, admin tier table
- Update README "Packages" sentence about tenancy
- CHANGELOG entry

### PR4 ‚Äî Security & ops hardening

Four independent items; can land as separate sub-PRs if you want finer control.

**(a) Webhook HMAC signatures**

- Schema: `ALTER TABLE agents ADD COLUMN webhook_secret TEXT` (nullable; auto-generated on first webhook config)
- `src/webhooks/sign.ts` (new) ‚Äî HMAC-SHA256 helper, timing-safe compare
- Every webhook dispatch adds `X-AgentStep-Signature: sha256=<hex>` header
- `docs/webhooks.md` ‚Äî signature format, example verifier code in JS + Python

**(b) Redis rate-limit backend**

- `packages/agent-sdk/src/auth/rate_limit.ts` grows a dispatcher: `RATE_LIMIT_BACKEND=memory` (today's default) vs `RATE_LIMIT_BACKEND=redis REDIS_URL=...`
- Redis impl uses `INCR` + `EXPIRE` ‚Äî classic fixed-window pattern that tolerates cluster scale
- Optional dependency: `ioredis` as peer dep ‚Äî only required when `RATE_LIMIT_BACKEND=redis`
- README: document the clustering story

**(c) Audit log**

- Schema: new `audit_log` table `(id, key_id, tenant_id, event, resource_type, resource_id, payload_json, created_at)`
- `src/auth/audit.ts` (new) ‚Äî `logAudit({auth, event, resource})` helper, called from every admin-only CRUD endpoint
- Events: `api_key.created`, `api_key.revoked`, `api_key.updated`, `tenant.created`, `upstream_key.added`, `upstream_key.disabled`, `agent.archived`, etc.
- `GET /v1/audit?tenant_id=...&event=...&from=...&to=...` admin-only
- UI: `/settings/audit` page (tenant admins see their own tenant's log; global admins see all)

**(d) CLI parity**

- `gateway api-keys create|list|get|revoke|patch` ‚Äî matching the HTTP API
- `gateway tenants ...` (already in PR1 plan)
- `gateway upstream-keys add|list|disable|delete`
- `gateway budgets reset <key-id>` ‚Äî admin helper for the monthly-reset story
- Output matches the `gateway agents`/`gateway sessions` command style

## Verification

1. **Schema idempotency** ‚Äî boot against a 0.4.0-seeded DB copy, no errors.
2. **PR1 ‚Äî tenant isolation**:
   - Create two tenants T1, T2. Create agent a1 in T1. Key k_T2 (tenant admin of T2) cannot list a1 (`GET /v1/agents` returns [], `GET /v1/agents/a1` returns 404).
   - Global admin key sees both tenants' agents unfiltered.
   - `gateway tenants migrate-legacy` on a freshly-upgraded 0.4.0 DB assigns all null-tenant rows to the default tenant.
   - `fallback_json` pointing to a cross-tenant agent is rejected at save (400) AND skipped at runtime even if previously-valid.
   - Tenant admin cannot create a key in a different tenant (400).
   - Seed key remains global admin across the upgrade (`tenantId=null`, `admin=true`).
3. **PR2 ‚Äî upstream pool multi-provider**:
   - Add OpenAI and Gemini keys to the pool alongside Anthropic. Create sessions on each engine; verify `last_used_at` rotates per-provider.
4. **PR3 ‚Äî upstream UI**:
   - Admin creates, disables, enables, deletes pool keys via the UI. Tenant admin on a non-upstream-admin key sees the section but gets a 403 banner.
5. **PR4a ‚Äî webhook signatures**:
   - Register a webhook. Trigger a session event. The delivered HTTP request has a valid `X-AgentStep-Signature` matching `sha256(body, secret)`.
6. **PR4b ‚Äî Redis rate limit**:
   - Boot two `gateway serve` processes sharing a Redis instance. Hit one process's rate limit via the other process's endpoint ‚Äî both should see the same counter.
7. **PR4c ‚Äî audit log**:
   - Create a virtual key via the API. `GET /v1/audit?event=api_key.created` returns the entry with the creating key_id and tenant_id.
8. **PR4d ‚Äî CLI parity**:
   - `gateway api-keys create --name bot --scope-agents agent_abc` prints the raw key; `gateway api-keys list` shows it.
9. **Test count**: baseline 560 + new ‚âà 640‚Äì680 total.
10. **Upgrade smoke**: take a 0.4.0 DB with live data, run `gateway serve` on 0.5.0. No errors. All existing sessions continue; new sessions attributable to tenants (if migrate-legacy has been run).

## Release

- Ship as **v0.5.0** (minor bump, backwards-compatible on read: unmigrated null-tenant rows remain visible to global admin; migration is opt-in).
- Breaking change gate: if a tenant admin tries to access a null-tenant resource, they get 404. Docs prominently recommend running `gateway tenants migrate-legacy` before granting non-global keys to anyone.
- CHANGELOG entry. Tag. Release workflow publishes `@agentstep/gateway@0.5.0` and `@agentstep/agent-sdk@0.5.0` to npm.

## Out of scope ‚Äî candidates for v0.6+

- Per-tenant upstream pools (`upstream_keys.tenant_id`)
- Tenant-scoped `memory_stores`, `files`, `proxy_resources`, `settings`
- Time-series metrics for `group_by=agent|environment|backend` (today it's api_key-only)
- Monthly budget auto-reset (cron)
- Configurable fallback depth
- Session tagging for cost allocation (`{project: "acme", team: "infra"}`)
- SSO / SAML / OIDC ‚Äî external IdP integration
- External SIEM export for the audit log
- Webhook retries with exponential backoff
- CLI-driven export/import of tenant state (for backup/migration)
