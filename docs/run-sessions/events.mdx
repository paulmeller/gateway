---
title: Session Event Stream
description: Send messages to a session, stream agent responses in real time, and handle custom tool calls — all through a single append-only event log.
---

Every interaction with a session goes through an **event log**: an append-only sequence of typed events shared between user and agent. You write `user.*` events to drive the agent; the server appends `agent.*`, `session.*`, and `span.*` events as the agent runs. A live SSE stream tails the log in real time.

## Send events

```http
POST /v1/sessions/:id/events
```

The body is a batch of one or more events:

```json
{
  "events": [
    {
      "type": "user.message",
      "content": [{ "type": "text", "text": "Refactor the auth module to use JWT." }]
    }
  ]
}
```

Multiple events in one batch are processed in order. To avoid duplicate processing on retries, include the `Idempotency-Key` header — the gateway appends `:0`, `:1`, etc. per event within the batch.

The response is the list of event rows that were persisted:

```json
{
  "data": [
    { "id": "evt_01...", "seq": 1, "session_id": "session_01...", "type": "user.message", ... }
  ]
}
```

## User event types

### `user.message`

Sends a text message to the agent. This starts a new turn if the session is `idle`.

```json
{
  "type": "user.message",
  "content": [{ "type": "text", "text": "Your message here" }]
}
```

`content` is an array of text blocks. Sending a message while a turn is already `running` queues it as pending input — it will be delivered to the next turn.

### `user.interrupt`

Signals the agent to stop the current turn as soon as possible.

```json
{ "type": "user.interrupt" }
```

The running process receives an interrupt signal. The session transitions to `idle` with `stop_reason: "interrupted"`. Any pending user input is discarded.

### `user.custom_tool_result`

Returns the result of a custom tool call. When the agent emits an `agent.custom_tool_use` event the session pauses and waits for this event before continuing.

```json
{
  "type": "user.custom_tool_result",
  "custom_tool_use_id": "toolu_01...",
  "content": [{ "type": "text", "text": "{ \"rows\": 42 }" }]
}
```

`content` is an array of content blocks. The `custom_tool_use_id` must match the `id` from the corresponding `agent.custom_tool_use` event.

### `user.tool_confirmation`

Approve or deny a tool call when the agent has `confirmation_mode` enabled. The agent pauses at each tool invocation until this event is received.

```json
{
  "type": "user.tool_confirmation",
  "tool_use_id": "toolu_01...",
  "result": "allow"
}
```

To deny and provide a reason:

```json
{
  "type": "user.tool_confirmation",
  "tool_use_id": "toolu_01...",
  "result": "deny",
  "deny_message": "Do not modify production files."
}
```

### `user.define_outcome`

Attaches a success rubric to the session. The agent uses this to self-evaluate and decide when to stop.

```json
{
  "type": "user.define_outcome",
  "description": "The auth module should use stateless JWT tokens with a 1-hour expiry.",
  "rubric": "All existing tests pass. No session cookies remain.",
  "max_iterations": 10
}
```

## Agent event types

These events are appended by the server as the agent runs. You receive them on the SSE stream.

### `agent.message`

A text response from the agent.

```json
{
  "type": "agent.message",
  "content": [{ "type": "text", "text": "I've refactored the module. Here's what changed..." }]
}
```

### `agent.thinking`

Internal reasoning text produced before a response (only emitted by models that support extended thinking).

```json
{
  "type": "agent.thinking",
  "thinking": "The caller wants JWT. I should update the middleware first..."
}
```

### `agent.tool_use`

A built-in tool call (Bash, Read, Write, Edit, Glob, Grep, WebFetch, WebSearch).

```json
{
  "type": "agent.tool_use",
  "id": "toolu_01...",
  "name": "Bash",
  "input": { "command": "npm test" }
}
```

### `agent.tool_result`

The result of a built-in tool call.

```json
{
  "type": "agent.tool_result",
  "tool_use_id": "toolu_01...",
  "content": [{ "type": "text", "text": "All 42 tests passed." }]
}
```

### `agent.custom_tool_use`

The agent is invoking a custom tool defined on the agent. The session pauses here until you send a `user.custom_tool_result`.

```json
{
  "type": "agent.custom_tool_use",
  "id": "toolu_01...",
  "name": "query_database",
  "input": { "sql": "SELECT COUNT(*) FROM users" }
}
```

## Session event types

### `session.status_running`

The agent has started processing a turn.

```json
{ "type": "session.status_running" }
```

### `session.status_idle`

The agent has finished the current turn.

```json
{
  "type": "session.status_idle",
  "stop_reason": "end_turn"
}
```

Common `stop_reason` values:

| Value | Meaning |
|-------|---------|
| `end_turn` | Agent completed its response normally |
| `error` | An unrecoverable error stopped the turn |
| `interrupted` | A `user.interrupt` was received |
| `custom_tool_call` | Paused waiting for a `user.custom_tool_result` |

### `session.status_terminated`

The session has been permanently closed (e.g. after a `DELETE /v1/sessions/:id` request).

```json
{
  "type": "session.status_terminated",
  "payload": { "reason": "deleted" }
}
```

### `session.error`

An error occurred during processing.

```json
{
  "type": "session.error",
  "error": { "message": "Container exited with code 137" }
}
```

## Span events

Span events carry OpenTelemetry-style trace context. Each event carries `trace_id`, `span_id`, and `parent_span_id` fields.

| Type | Description |
|------|-------------|
| `span.model_request_start` | The model API call started |
| `span.model_request_end` | The model API call completed; carries token usage |

```json
{
  "type": "span.model_request_end",
  "trace_id": "abc123",
  "span_id": "def456",
  "model_usage": {
    "input_tokens": 2100,
    "output_tokens": 420,
    "cache_read_input_tokens": 1800,
    "cache_creation_input_tokens": 0
  }
}
```

## Stream events over SSE

```http
GET /v1/sessions/:id/stream
```

Opens a Server-Sent Events connection that delivers events in real time. Each SSE message carries the event type and the full JSON event object:

```
id: 5
event: agent.message
data: {"id":"evt_01...","seq":5,"type":"agent.message","content":[{"type":"text","text":"Done."}]}

id: 6
event: session.status_idle
data: {"id":"evt_01...","seq":6,"type":"session.status_idle","stop_reason":"end_turn"}
```

The gateway sends a keepalive every 15 seconds:

```
data: {"type":"ping"}
```

### Reconnecting

On reconnect, pass the `Last-Event-ID` header with the last `seq` you received, or the `after_seq` query parameter. The stream will replay any events you missed before switching to live delivery:

```http
GET /v1/sessions/:id/stream?after_seq=5
Last-Event-ID: 5
```

The stream deduplicates events between the backlog drain and the live tail to prevent double delivery during the small reconnect window.

## List past events

```http
GET /v1/sessions/:id/events
```

Returns a paginated list of all events from the session's log.

| Parameter | Type | Description |
|-----------|------|-------------|
| `limit` | number | Events per page (default 50) |
| `order` | `asc` \| `desc` | Sort by sequence number (default `asc`) |
| `after_seq` | number | Return only events with `seq` greater than this value |
| `page` | string | Alias for `after_seq` |

```json
{
  "data": [
    { "id": "evt_01...", "seq": 1, "type": "user.message", ... },
    { "id": "evt_02...", "seq": 2, "type": "session.status_running", ... }
  ],
  "next_page": "2"
}
```

Pass `next_page` as the `page` parameter to fetch the next batch.

## Handling custom tool calls

When an agent has a custom tool defined, the session enters a requires-action state at `agent.custom_tool_use`. Your integration loop should:

1. Stream events until you receive `session.status_idle` with `stop_reason: "custom_tool_call"`.
2. Find the preceding `agent.custom_tool_use` event and extract `id` and `input`.
3. Execute your tool locally.
4. POST a `user.custom_tool_result` with the output.
5. The agent immediately resumes the turn.

```typescript
// Minimal custom tool loop
const stream = openSSE(`/v1/sessions/${sessionId}/stream`);

for await (const event of stream) {
  if (event.type === "session.status_idle" && event.stop_reason === "custom_tool_call") {
    // Find the matching tool_use in recent events
    const toolUse = await getLastCustomToolUse(sessionId);
    const result = await myTool(toolUse.name, toolUse.input);

    await postEvents(sessionId, [{
      type: "user.custom_tool_result",
      custom_tool_use_id: toolUse.id,
      content: [{ type: "text", text: JSON.stringify(result) }],
    }]);
  }

  if (event.type === "session.status_idle" && event.stop_reason === "end_turn") {
    break;
  }
}
```

The gateway handles the `memory` built-in tool server-side — you do not need to implement that one yourself.
