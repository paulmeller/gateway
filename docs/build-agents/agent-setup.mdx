title: Define Your Agent
description: Agent config fields, create/update/archive lifecycle, and integer versioning with optimistic concurrency.
---

An agent is the central configuration object: it defines which model and engine runs, what the system prompt says, which tools are available, and how webhooks and skills are attached. Every mutation to an agent creates a new immutable version row, so sessions always reference a reproducible snapshot.

## Create an agent

```http
POST /anthropic/v1/agents
```

```json
{
  "name": "code-reviewer",
  "model": { "id": "claude-opus-4-5" },
  "engine": "claude",
  "description": "Reviews diffs and outputs structured feedback.",
  "system": "You are a senior engineer. Review the diff and output structured feedback.",
  "tools": [
    {
      "type": "agent_toolset_20260401",
      "default_config": { "enabled": true },
      "configs": [
        { "name": "WebSearch", "enabled": false }
      ]
    }
  ]
}
```

The response body is an `Agent` object at `version: 1` with `type: "agent"`.

## Config fields

| Field | Type | Default | Description |
|---|---|---|---|
| `name` | string | — | Display name. Must be unique within your tenant. |
| `model` | `string \| { id: string, speed?: "standard" \| "fast" }` | — | Model identifier. Pass as bare string (`"claude-sonnet-4-6"`) or object (`{ id: "...", speed: "fast" }`). Engine is auto-inferred from model prefix if `engine` is omitted. |
| `engine` | string | auto-inferred | Execution engine: `claude`, `opencode`, `codex`, `gemini`, `factory`, `pi`, or `anthropic`. Auto-inferred from model prefix if omitted (`gemini-*` → `gemini`, `gpt-*`/`o1-*` → `codex`, `claude-*` → `claude`). |
| `description` | string \| null | `null` | Human-readable description (max 2048 characters). |
| `system` | string \| null | `null` | System prompt injected before every turn. |
| `metadata` | Record\<string, string\> | `{}` | Arbitrary key-value pairs (max 16 pairs). |
| `tools` | array | `[{type:"agent_toolset_20260401"}]` | Tool configuration. See [Tools](/build-agents/tools). |
| `mcp_servers` | array | `[]` | MCP server declarations. Each entry is `{ name: string, type: "url", url: string }`. See [MCP Connector](/build-agents/mcp-connector). |
| `skills` | array | `[]` | Skill bundles installed into the container at session start. See [Skills](/build-agents/skills). |
| `webhook_url` | string \| null | `null` | URL that receives session lifecycle events. |
| `webhook_events` | string[] | `["session.status_idle","session.status_running","session.error"]` | Which events to deliver. |
| `webhook_secret` | string | — | HMAC-SHA256 shared secret (min 32 chars). When set, deliveries include `X-AgentStep-Signature: sha256=<hex>`. The secret is never returned by the API — only `webhook_signing_enabled: true/false` is surfaced. |
| `multiagent` | object \| null | `null` | Multi-agent coordinator config. Set to `{ type: "coordinator", agents: [...] }` to enable thread-based delegation. See [Multi-Agent](/build-agents/multi-agent). |
| `threads_enabled` | boolean | `false` | Adds a synthetic `spawn_agent` custom tool so the agent can delegate sub-tasks. |
| `confirmation_mode` | boolean | `false` | Enables the permission bridge. Claude runs with `default` permission mode and pauses before destructive tool calls. |
| `permission_policy` | object \| null | `null` | Per-tool permission control. `always_allow` lists tools that run without confirmation; `always_ask` lists tools that require explicit approval via `user.tool_confirmation` events. See below. |
| `callable_agents` | array | `[]` | Agents this agent may spawn when `threads_enabled` is `true`. Each entry is `{type:"agent", id, version?}`. |
| `model_config` | object | `{}` | Engine-level model hints (deprecated in favor of `model.speed`). |

### Engine constraints

- **Auto-inference**: When `engine` is omitted, it's inferred from the model prefix: `gemini-*` → `gemini`, `gpt-*`/`o1-*`/`o3-*`/`codex-*` → `codex`, `claude-*` → `claude`. Unknown prefixes default to `claude`. You can always override by setting `engine` explicitly.
- `tools` may only be set on the `claude` engine. Other engines manage their own tool permissions internally — passing `tools` to a non-claude agent returns a 400.
- `model` is validated against the engine's known model list at creation time. An unsupported model returns a 400 with the list of valid models.
- `engine` cannot be changed after creation.

### Vault keys per engine

Each engine requires its own API key in the vault:

| Engine | Vault key | CLI installed in container |
|--------|-----------|--------------------------|
| `claude` | `ANTHROPIC_API_KEY` | `@anthropic-ai/claude-code` |
| `gemini` | `GEMINI_API_KEY` | `@google/gemini-cli` |
| `codex` | `OPENAI_API_KEY` or `CODEX_API_KEY` | `@openai/codex` |

### Permission policies

Control per-tool permissions instead of the all-or-nothing `confirmation_mode`:

```json
{
  "name": "safe-agent",
  "model": "claude-sonnet-4-6",
  "permission_policy": {
    "always_allow": ["Read", "Grep", "Glob", "WebSearch"],
    "always_ask": ["Write", "Edit", "Bash"]
  }
}
```

- `always_allow` — tools that run without confirmation
- `always_ask` — tools that pause and emit `agent.tool_confirmation_request`; client must respond with `user.tool_confirmation` (allow/deny)
- Tools not in either list follow the engine's default behavior
- Setting `permission_policy` automatically enables `--permission-mode default` on the Claude CLI

## Update an agent

```http
PATCH /anthropic/v1/agents/:id
```

Send only the fields you want to change plus the current `version` for optimistic concurrency. Every successful PATCH increments `version` and writes a new `agent_versions` row. The previous version remains readable via the `?version=N` query parameter on GET. If the `version` in the request body does not match the agent's current version, the server returns `409 Conflict`.

```json
{
  "version": 1,
  "model": { "id": "claude-sonnet-4-5" },
  "system": "Updated system prompt."
}
```

Passing `webhook_secret: null` clears the secret so subsequent deliveries are unsigned. Omitting the field leaves the existing secret untouched.

## Agent versioning

Every agent carries an integer `version` counter. The response from GET, POST, and PATCH always returns the current (highest) version. To retrieve a historical snapshot:

```http
GET /anthropic/v1/agents/:id?version=2
```

Versions are written in a transaction that increments `current_version` on the parent `agents` row, so concurrent updates are serialized — the second writer will see the latest state and should retry if needed.

Existing sessions pin the `agent_version` they were created with. A PATCH to the agent does not affect running or idle sessions.

## List agents

```http
GET /anthropic/v1/agents
```

Query parameters:

| Parameter | Description |
|---|---|
| `limit` | Number of results (1–100, default 20). |
| `order` | `asc` or `desc` (default `desc`). |
| `page` | Cursor — the `id` of the last result from the previous page. |
| `include_archived` | `true` to include archived agents. |

Returns `{ data: Agent[], next_page: string | null }`.

## Archive an agent

```http
POST /anthropic/v1/agents/:id/archive
```

Sets `archived_at` and returns the full agent object. Archived agents are excluded from list results unless `include_archived=true`. All historical version rows are preserved; sessions that reference the agent continue to function.

Archiving is irreversible through the API — there is no unarchive endpoint.

The legacy `DELETE /anthropic/v1/agents/:id` endpoint is still supported and returns `{ id, type: "agent_deleted" }`.

## List agent versions

```http
GET /anthropic/v1/agents/:id/versions
```

Returns a paginated list of all version snapshots for the agent. Each entry is a full agent object at that version. Pagination uses `{ data, next_page }` — pass `next_page` as the `page` query parameter to fetch the next batch.
